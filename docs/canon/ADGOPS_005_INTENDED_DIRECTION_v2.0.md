## 0. What this document is

This is the intended-direction document for ADG OPS wave-2.

It exists to answer:
- what ADG OPS is trying to become
- what future direction is already credible
- what kinds of changes are likely
- what present-day work should be preparing for
- how the current suite bridges into future capability

This is not:
- a speculative wishlist
- the full roadmap
- a patch request
- a manifesto detached from repo truth
- a permission slip for autonomous redesign

One-line doctrine:

**ADG OPS should evolve deliberately from its current real suite into a broader intelligence ecosystem without losing traceability, learnability, or product control.**

---

## 1. Directional reading

ADG OPS is not only a current suite of surfaces.
It is also a system being prepared for a broader operational future.

The intended future includes:
- stronger data intake
- stronger structured storage
- better sector intelligence
- broader ecosystem-facing services
- more operational depth across the suite

This future is real enough to document.
It is not yet real enough to be described as current implementation truth.

---

## 2. Core directional bridge

The intended direction must always be read through two anchors:

### 2.1 Current anchor
- current repo truth
- current surfaces
- current runtime reality
- current blockers and fragility

### 2.2 Future anchor
- the suite becoming broader, deeper, and more operationally useful
- architecture being prepared for later additions
- current code being shaped so future growth does not force wasteful rewrites

Rule:
- present work must remain traceable to real current needs
- but the system should still be shaped so future additions can land cleanly later

---

## 3. Intended future of the suite

ADG OPS is moving toward a broader ecosystem that can support:

- public procurement intelligence
- configurable statistical reading
- period-based interpretive analysis
- improved data intake and document ingestion
- structured storage and richer querying
- operational support services
- ecosystem-oriented modules such as directories, trackers, and alerts

This is not a claim that all of these are equally mature now.
It is a directional system reading.

---

## 4. Likely future change areas

The following areas are likely to change materially over time:

### 4.1 Data architecture
The current data path is not the final one.
The system is likely to move toward:
- more than one fetcher
- richer ingestion
- better separation of intake roles
- stronger structured storage
- future database integration

### 4.2 Intelligence depth
The product is likely to evolve from:
- surface-level reading
toward:
- richer sector interpretation
- better cross-period understanding
- stronger ecosystem explainability

### 4.3 Service layer
Some currently lighter or secondary surfaces may later become more central because:
- alerts may gain more weight
- directories or opportunity layers may become core utilities
- supporting ecosystem services may become part of normal product use

### 4.4 Operational robustness
The direction is not only “more features.”
It is also:
- better workforce operation intelligence
- pruning what blocks progress
- updating intelligence and work method
- consolidating the system so it can be finished well

This should be read as directional context, not as permission to improvise implementation.

---

## 5. Directional sequencing logic

The intended future should be approached through a probable sequence:

### 5.1 Stabilize current truth
- clarify what the suite is today
- fix critical blockers
- document runtime truth honestly

### 5.2 Prepare structure for growth
- make current areas logic-ready
- reduce ambiguity in ownership
- avoid changes that would trap later additions

### 5.3 Expand ecosystem capability
- deeper data intake
- stronger structured data handling
- broader service layer
- future modules that support the wider ecosystem

This is not a frozen roadmap.
It is the intended sequencing logic that should help evaluate present-day decisions.

---

## 6. Current-to-future doctrine

ADG OPS should not be developed under either of these bad extremes:

### Bad extreme A
Build only for what exists right now, with no regard for later architecture.

### Bad extreme B
Build for an imagined future and lose contact with the current real system.

Correct rule:
- current truth must govern current implementation
- intended direction must inform structural readiness
- current changes should be explainable in both directions:
  - why they solve today’s need
  - why they do not sabotage tomorrow’s system

---

## 7. Directional constraints for coding partners

Claude or any coding LLM should operate inside this intended-direction layer as:

- senior coding partner
- simplifier
- technical challenger when appropriate

Allowed:
- “this can be done in 50 lines instead of 500”
- “this approach reaches the same goal more cleanly”
- “this current proposal creates unnecessary technical burden”

Not allowed:
- changing product direction
- redefining module meaning
- silently altering user-defined product behavior
- replacing traceable work with autonomous invention

Rule:
- every meaningful change must still map back to a real problem, real constraint, or real intended capability
- if something is unclear, questions must be asked instead of guessed

---

## 8. Human + LLM partnership doctrine

The intended future assumes a working model where LLMs are treated as:
- serious working partners
- not magical oracles
- not passive output machines
- not owners of product direction

The human side must retain:
- judgment
- learning capacity
- operational understanding
- product ownership

The coding partner side may contribute:
- simplification
- technical guidance
- implementation craft
- challenge against inefficient technical paths

This matters because over-delegation weakens both product control and learning.

---

## 9. What is intentionally not frozen yet

The following are directionally real but not yet fully frozen:

- exact DB/SQL integration shape
- final data model for broader suite growth
- exact service boundaries for alerts/directories/opportunities
- exact multi-fetcher architecture
- exact sequencing of all future modules

These areas should remain:
- documented
- traceable
- architecture-aware
- but not falsely locked before the project is ready

---

## 10. Locked directional conclusions

- ADG OPS is intended to grow beyond its current surface set
- current work must stay traceable to real present needs
- future readiness matters, but must not justify speculative overbuilding
- structured storage / SQL is an intended capability, still formative
- the data/fetching layer will likely broaden beyond the current fetcher
- supporting surfaces may later become more central
- coding partners may simplify technical paths, but may not redefine product meaning

---

## 11. One-line doctrine

**Prepare ADG OPS for the future by shaping today’s work intelligently, not by pretending tomorrow already exists.**
