## 0. What this document is

This is the future-modules document for ADG OPS wave-2.

It exists to answer:
- which future additions are already credible
- what type of addition each one is
- how mature each future direction currently is
- how those additions fit the suite
- which of them are already represented by stubs, placeholders, or conceptual preparation

This is not:
- a random backlog
- an implementation order doc
- a speculative idea dump
- the runtime-truth doc

One-line doctrine:

**Only future additions with real strategic weight should enter canon, and each one should be typed and maturity-marked clearly.**

---

## 1. Classification model

Each future addition should be classified by:

- **type**
  - module
  - service
  - ecosystem capability

- **maturity**
  - confirmed
  - strong candidate
  - early-forming

- **current state**
  - not yet present
  - conceptually present
  - stubbed / placeholder
  - partially active

This prevents:
- treating stubs as mature modules
- treating ideas as if they were already planned systems
- flattening all future work into one list

## 3. Important reading rules

### 3.1 Not all future additions are equal
Some are:
- closer to the core architecture
- more likely to shape current technical decisions

Others are:
- important
- but further away from present coding reality

### 3.2 Future additions should affect readiness, not create overbuilding
A future module belongs here if it changes:
- how we think about current architecture
- how we shape current surfaces
- how we avoid future rewrites

It does not belong here just because it sounds interesting.

### 3.3 Stubbed or placeholder presence must stay explicit
If something exists only as:
- a stub
- a placeholder
- a conceptual preparation layer

that must stay visible.
No future module should be narrated as “already real” unless runtime truth supports it.

---

## 4. Priority reading

### Highest architectural impact
- structured data / SQL layer
- expanded fetching architecture
- PDF / deeper ingestion

### High ecosystem impact
- email alerts
- partner directory
- Laus Tracker

### Strong utility expansion
- internship board

This is not final implementation order.
It is a reading of likely future weight.

---

## 5. Relationship to current suite

The future modules layer should be read as:

- extending the current suite
- not replacing it
- not invalidating present surface work
- not authorizing speculative architecture explosion

Current surfaces still matter because they are the living base the future system will build on.

---

## 6. Relationship to coding partners

Coding partners may:
- simplify technical paths
- recommend better structures
- point out overbuilt directions

But they may not:
- silently promote weak ideas into canon
- turn future module notes into implementation scope automatically
- replace product ownership with technical convenience

Questions should be asked when:
- maturity is unclear
- module vs service vs capability is unclear
- present implementation implications are unclear

---

## 7. Locked conclusions

- future additions should be grouped by domain and maturity
- type must stay visible: module / service / ecosystem capability
- SQL / structured storage is a real intended capability, still formative
- expanded fetching is a real future architecture line
- PDF ingestion is more than a loose note; it belongs in the intended data direction
- alerts, directory, tracker, and opportunities belong in canon as future-facing additions, not as current stable modules

---

## 8. One-line doctrine

**Future additions enter ADG OPS canon only when they are structurally relevant enough to shape how today’s system should be prepared.**
