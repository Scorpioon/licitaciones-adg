## 0. Purpose

This document preserves the closed product-audit ficha for Laus Tracker.

It exists to preserve:
- intended role
- editorial/non-competitive design doctrine
- data assumptions
- future analytical scope
- MVP vs strong-version boundary

---

## 1. One-line doctrine

**Laus Tracker is an editorial, historical, and expository archive of the ADG Laus awards, enriched with statistics and medallero-style views, but explicitly not designed as a competitive ranking product.**

---

## 2. Core positioning

Laus Tracker should be understood as:
- historical archive
- editorial showcase
- expository statistics surface
- sector memory layer
- awards-history exploration module
- future analytical layer for Laus history

It should not be understood as:
- competition engine
- scoreboard product
- public "who is better" ranking system
- gamified league table
- status hierarchy generator

---

## 3. Hard editorial rule

The product may contain:
- medallero
- rankings
- statistics
- evolution over time
- school rankings
- studio/author history
- category comparisons

But the experience must communicate:

**This is historical and editorial interpretation, not competitive scoring.**

Avoid:
- sports standings
- winner/loser framing
- aggressive ranking culture
- trophy-heavy hierarchy
- "top winners" as emotional product hook

---

## 4. Strategic role

Current:
- intended module
- waiting for data
- not implemented as working module

Future:
- historical LAUS archive
- editorial analytics layer
- possible pillar of broader ADG platform / LAUS platform replacement

This is intended direction, not current truth.

---

## 5. User groups

The module should serve:
- ADG / internal team
- studios and agencies
- schools
- students / talents
- broader design-sector public

Different users may care about different entry points, but the product should not split into competitive modes.

---

## 6. Data atom and historical structure

Likely LAUS structure:
1. Year / edition
2. Gran Laus awards
3. Categories
4. Subcategories
5. Award level:
   - Gold
   - Silver
   - Bronze
   - In Book
6. Project name
7. Studio / author
8. For student/talent entries:
   - teacher / tutor
   - school

Practical product atom:
**Awarded project within a year/category/subcategory/award-level context.**

---

## 7. Navigation model

Preferred navigation:
- by year
- by category
- by studio / author
- by school
- by award type
- by medallero view
- by editorial exploration hub

---

## 8. Medallero model

Confirmed:
- weighted by award type
- filterable by year
- filterable by category
- possible studio/school/author filters
- editorial/non-competitive framing

---

## 9. Data contract

### MVP

Required:
- year
- category
- subcategory where applicable
- award level
- project name
- studio / author
- school if applicable
- teacher / tutor if applicable

### Strong version

Add:
- images
- links
- project description
- jury
- credits
- metadata for people/studios/schools
- richer category taxonomy
- archival source references

Rule:
**No detailed UI or implementation claim before data contract confirmation.**

---

## 10. Visibility while waiting for data

Production:
- keep internal or disabled until data exists
- optional grey disabled stub in hub/home

Demo:
- mock data allowed only if clearly marked as demo
- mock data must not look like real LAUS historical truth

---

## 11. Risks

- competitive misreading
- incomplete historical data
- category taxonomy changes over time
- school/tutor data inconsistency
- mock data confusion
