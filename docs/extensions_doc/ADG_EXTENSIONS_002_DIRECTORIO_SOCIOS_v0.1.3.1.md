## 0. Purpose

This document preserves and clarifies the closed ficha for Directorio de Socios.

It defines:
- intended role
- current reality of the existing basic directory
- difference between legacy list and future module
- relationship to map, profiles, specialties, LAUS, and future community/forum layer

---

## 1. One-line current truth

**There is a legacy/basic directory, but not yet the full ADG Extensions directory module.**

---

## 2. Extension identity

Name:
- Directorio de Socios

Family:
- ADG Extensions

Type:
- ecosystem capability
- directory module
- future community/profile layer

Current state:
- Legacy / partial exists

Correction:
- a basic directory currently exists as a long page/list of names
- the intended navigable, searchable, profile-based directory does not yet exist in full

---

## 3. Main role

The module should become:
- public/professional ADG member directory
- professional community map
- discovery and navigation surface
- structured way to browse, sort, search, and understand the member network

---

## 4. Product spirit

Spirit:
- mixed, but community/ecosystem dominant
- avoid aggressive marketplace behavior
- avoid lead-generation machine framing

Doctrine:
**The directory should feel like ADG community infrastructure, not a commercial marketplace.**

---

## 5. Profile types

Included:
- studios
- freelancers
- agencies
- schools

---

## 6. Privacy / profile claim model

Future direction:
- public base listing can exist from directory data
- richer profiles should be claimable/editable through controlled flow
- profile claiming is future-critical but not MVP by default

Claimed profiles require:
- identity
- moderation
- permission
- ownership model
- opt-in for richer public/community features

---

## 7. Location model

Progressive location support:
- city
- province
- autonomous community
- country
- map-based navigation when useful

---

## 8. Specialty / tag model

Model:
- closed specialty list
- secondary tags later
- oldschool/newschool tag sets available
- LAUS taxonomy may inform but must not blindly define specialties

Rule:
**Closed specialty list first; secondary tags later; LAUS taxonomy referenced, not blindly inherited.**

---

## 9. UI model

Desktop:
- may use split/hybrid model
- map and cards can work together
- directory must not become just a map split view

Mobile:
- list/cards first
- search first
- results-focused flow

Doctrine:
**The directory is its own browsing/search surface; the map is a related spatial mode, not the whole product.**

---

## 10. Relationship to existing map

The map already exists as a surface.

Directory may:
- connect to map
- expand into map
- allow spatial exploration

But:
- directory must stand on its own
- user must be able to browse, sort, search, and navigate members without being forced into map-first behavior

---

## 11. Relationship to LAUS Tracker

Intended only if data match is reliable:
- member profiles may show LAUS history
- LAUS Tracker may link back to member profile
- school profiles may connect to student/talent LAUS history

No fuzzy matching should be shown as verified truth.

---

## 12. CTA model

Primary:
- View profile

Secondary:
- external web

Avoid:
- aggressive "Contact now"
- marketplace-style conversion pressure
- lead-gen framing

---

## 13. Data contract

MVP:
- name
- profile type
- location
- specialties
- web/contact
- short description

Strong version:
- social links
- projects
- LAUS history if matched
- profile image/logo
- claimed profile state
- public/private visibility
- update/moderation status
- community participation metadata

---

## 14. Future community/forum direction

Preserved future layer:
- claimable profiles
- lowkey old-school forum
- initial topics
- posts
- replies
- community layer

Classification:
- future community layer
- strategically relevant
- not MVP directory scope

Hard rule:
**The forum direction is real enough to preserve, but not current implementation scope.**

---

## 15. MVP boundary

MVP should focus on:
- searchable member directory
- profile type filters
- specialty filters
- location filters
- card/list browsing
- profile detail view
- basic relation to map
- data contract discipline

MVP should not include by default:
- live account system
- profile claiming flow
- posting
- replies
- forum moderation
- LAUS linking unless reliable
- advanced community behavior

---

## 16. Strong version boundary

Strong version may include:
- claimable profiles
- verified member management
- richer profile pages
- map integration
- LAUS history
- project links
- community topics
- member posts
- replies
- moderation/admin tools
- ecosystem graph reading

---

## 17. Risks

- turning directory into marketplace
- reducing directory to a map
- exposing data without permission model
- profile claiming without identity/moderation
- chaotic tag taxonomy
- false LAUS matching
- forum scope explosion
- confusing legacy directory with future module
